数据加密

算法 简介
BASE64 一种基于64个可打印字符表示二进制数据的方法，主要为了将二进制数据变成可见字符以便传输，可以解密
MD5 一种基于64个可打印字符表示二进制数据的方法，主要为了将二进制数据变成可见字符以便传输，可以解密
DES 一种对称加密算法，用于对二进制数据进行加密，明文、密文和秘钥的分组长度都是64
AES 密码学中的高级加密标准，本质上是一种对称分组密码体制，用来代替DES
RSA 一种非对称加密算法，存在一个公钥和一个私钥，分别用于加密和解密，速度比对称加密慢很多

对称加密：加密解密使用同一秘钥

非对称加密：存在一个公钥和一个私钥

本地存储加密

本地存储方式：NSUserDefaults、文件、数据库

存储数据时，可以使用哈希算法、对称加密算法（DES/AES）进行加密存储，或者系统 keychain 进行存储，实现加密算法时也不能将 key 以明文形式写在代码中

网络传输加密

HTTPS + 本地证书校验方式

传输过程中可以使用 RSA+AES 进行加密传输

  1. 客户端随机生成 AES 的key
  2. 使用 RSA 公钥加密算法加密随机的 key
  3. 使用随机的 key 对数据进行 AES 加密
  4. 将 RSA 加密后的 key 和 AES 加密后的数据发送给服务器
  5. 给发送的数据加上时间戳，以防止重放攻击

反注入

RESTRICT

注入的方式通常使用 DYLD_INSERT_LIBRARIES 环境变量将 dylib 注入

自己编写的 dylib 无法注入到目标应用的时候，表明应用做了反注入保护。

  • iOS10以下系统（iOS10 以后 苹果的 DYLD 不再检测 RESTRICT 字段了）

在Xcode的编译设置选项 Other Linker Flags 添加 -Wl,-sectcreate,__RESTRICT,__restrict,/dev/null,

APP 运行后 MachOView 查看可执行文件可以看到多了一个 Section64(__RESTRICT,__restrict)

dyld 检测到 MachO中存在 __RESTRICT,__restrict 这样的节（Section）时，DYLD_INSERT_LIBRARIES 环境变量会被忽略，导致注入失败。

  • 反反注入（RESTRICT防护破坏）：

1、使用二进制修改工具 Synalyze+It_1.22_xclient.info/010Editor，直接搜索 restrict 再修改对应二进制保存

2、Load Command - LC_SEGMENT_64（__RESTRICT）修改 Data 值

3、使用Cydia自带的cynject工具注入，可以无惧主程序完整性检测 /usr/bin/cynject<pid><dylib>

Hook检测

iOS 上的 Hook 方式无外乎两种，Method Swizzing 和 Inline Hook

Method Swizzing 检测

Method Swizzing 原理是替换 IMP，通过 dladdr 得到 IMP 地址所在模块信息进行判断

Inline Hook检测

Inline Hook原理通过修改函数的前 N字节内存，使程序跳转到自己编写的函数

对函数头几个字节进行检测，以确定是否被 Hook

越狱检测

检测越狱商店及附属文件
syscall和汇编判断文件

syscall函数可以用来发送系统调用号来获取文件状态，从而判断文件是否存在，比如发送 fstatat64 调用号就相当于调用了 fstatat 函数

为了防止 syscall 函数被 Hook，可以直接调用 SVC 汇编指令来发送系统调用号

尝试读取系统应用列表

如果已越狱，就可以读取到系统应用列表

检查动态库列表

越狱过的系统一般都会加载 MobileSubstrate.dylib 动态库，动态库列表中如果包含 MobileSubstrate.dylib 动态库，那就是越狱了，调用 _dyld_get_image_name 获取动态库列表

检测URLScheme是否有效
检查函数是否被劫持

stat 函数出自 libsystem_kernel.dylib ，如果检查到了并非出自 libsystem_kernel.dylib 的函数，那肯定被劫持了，说明系统是越狱过的

dladdr 函数判断了 stat 函数的出处。

dladdr 函数也可能被 Hook。针对这种情况，可以读取 dladdr 函数的内存数据，判断是否被 Hook

MSHookFunction 会修改函数的前几个字节，因此只要对比前几个字节就能知道函数是否被 Hook，看前 4 字节就能准确识别函数是否被 Hook

比如 64 位函数前 4 个字节如果是 50000058，或者 32 位函数的前 4 字节如果 dff800f0 就说明使用了 MSHookFunction 对目标函数做了 Hook

自定义一个 isMSHookFunction 函数，两个参数，一个是动态库路径，一个是函数的名称，通过 dlopen 和 dlsym 得到 stat 函数的地址，然后读取内存，并判断开头 4 个字节的内容

完整性检测

检测目标是否被脱壳、是否被静态注入、内容是否被修改、是否被重签名等

常见方式有加载命令检测、代码段检测、签名信息检测

加载命令检测

为了使未越狱设备也能顺利使用插件，通常会在 LoadCommand（加载命令）中增加一个 LC_LOAD_DYLIB 或者 LC_LOAD_WEAK_DYLIB 的方式来指向自己的动态库，从而实现注入功能，这种方式通常也称为静态注入

加载命令检测即检测 Mach-O 文件的 LoadCommand 中 LC_LOAD_DYLIB 或 LC_LOAD_WEAK_DYLIB 是否存在异常

代码段检测

检测 __TEXT,__text 是否被修改（当然也可以检测其它 Section）

&_mh_execute_header 指向 MachO 在内存中的起始位置，getsectiondata返回指定 Section 的内存地址大小。实际应用中可以将正确的 Hash 值存到 服务器或者加密后放到本地，在进程启动时进行对比，不匹配则可以进行一些逻辑处理

签名信息检测

检测 LC_CODE_SIGNATURE 中的特征信息，比如 TeamID是否和用户匹配

检查Bundle identifier
检查App Store来源

判断文件格式中 Load commands 里的 LC_ENCRYPTION_INFO 和 LC_ENCRYPTION_INFO_64 两个信息是否被加密，如果是加密的则是 App Store 下载的

检查重签名

签名机制使用应用目录上的 embedded.mobileprovision 配置文件，每个配置文件中都会有一个 UUID 和账户的前缀，通过校验这两个来判断是否重签名

代码混淆

inline 内联函数

ARM 汇编里，函数调用的原理就是使用 bl 指令跳转到指定的函数地址，使得攻击者很容易就能通过逆向分析找到调用过程并分析参数和返回值

可以将一些敏感函数（比如加密函数）写为 inline 内联，这样无论代码调用函数多少次，编译器都会将整个函数代码复制到需要调用的地方，而不会使用 bl 指令跳转，这样就无法明显地看出函数的参数传递过程和返回值

防止被抓包

检测HTTP代理

通过获取 iOS 系统设置的代理信息可以检测出有没有开启代理，如果开启了代理可以不请求网络，或者断开连接

获取代理信息的方法是调用 CFNetworkCopySystemProxySettings 函数

这种检测方式容易被绕过，Hook CFNetworkCopySystemProxySettings 函数，对返回的字典进行修改即可。

为了防止这一点，可以直接采取读取配置文件的方式来获取代理，代理信息保存在 /Library/Preferences/SystemConfiguration/preferences.plist 文件里，遍历文件查找即可。DeviceName 为en0表示无线网络
